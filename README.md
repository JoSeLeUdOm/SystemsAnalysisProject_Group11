# Local Volunteer Coordination Platform
> A digital platform to connect volunteers with academic support projects in local communities of Bogotá, D.C., Colombia.

## 📋 Project Description

This platform optimizes the assignment of volunteers to tutoring and educational support projects aimed at children and adolescents in vulnerable communities. Through a weighted compatibility algorithm, the system connects the supply of volunteer talent with the demand for academic support, transforming a manual and disorganized process into an efficient, traceable digital system.

## 🎯 System Objectives

- **Reduce assignment times** between volunteers and educational institutions
- **Improve compatibility** between volunteer skills and project needs
- **Ensure effective schedule coordination** without availability conflicts
- **Measure the academic impact** generated by tutoring sessions
- Contribute to **reducing school dropout rates** in vulnerable communities

## 🧠 Compatibility Algorithm

The core of the system is an algorithm that calculates a composite compatibility score `C(v, p)` for each volunteer–project pair:

```
C(v, p) = w1·Skill(v,p) + w2·Schedule(v,p) + w3·Location(v,p) + w4·Experience(v,p)
```

| Component | Description | Default Weight | Data Source |
|---|---|---|---|
| `Skill(v,p)` | Overlap between volunteer subject competencies and project academic needs | **w1 = 0.40** | Survey P2 + Project registration |
| `Schedule(v,p)` | Percentage of required time slots covered by the volunteer's availability | **w2 = 0.30** | Survey P5, P6 |
| `Location(v,p)` | Geographic proximity score (in-person) or connectivity quality score (virtual) | **w3 = 0.15** | Survey P7 + Institution address |
| `Experience(v,p)` | Pedagogical experience score mapped to student grade level requirements | **w4 = 0.15** | Survey P4 |

## 🏗️ System Architecture

The system is structured in **three tiers** to ensure separation of responsibilities, independent scalability, and modular maintenance:

```
┌─────────────────────────────────────────────────────────┐
│                  PRESENTATION LAYER                     │
│  Volunteer Portal │ Institution Portal │ Admin Dashboard│
│                   Reports & Analytics                   │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────┐
│                   APPLICATION LAYER                     │
│  Volunteer Registration │ Institution Registration      │
│  Matching Algorithm │ Scheduling Engine                 │
│                    Impact Monitor                       │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────┐
│                      DATA LAYER                         │
│  Volunteer Profiles DB │ Projects & Schedules DB        │
│  Session Log DB │ Analytics Repository                  │
│          (Google Sheets API — initial phase)            │
└─────────────────────────────────────────────────────────┘
```

> **Note:** Given the initial scope of the project, **Google Sheets will be used as a temporary database** via the Google Sheets API, which allows reading and writing data from Python.

## 🛡️ Risk Management

| Sensitivity Factor | Reference | Mitigation Mechanism |
|---|---|---|
| Variability in volunteer supply | Workshop 1 - Sec. 3.2 | Waiting queue and temporary matching assignments |
| Scheduling conflicts | Workshop 1 - Sec. 3.2 | Conflict detection before session confirmation |
| Low connectivity in communities | Workshop 1 - Sec. 2.5 | App with session recording in offline mode |
| Changes in student demand | Workshop 1 - Sec. 3.4 | Weekly/monthly reports during demand peaks |

## 📊 Performance Metrics

| Metric | Target |
|---|---|
| Volunteer–student matching response time | < 1 minute |
| System availability (monthly) | > 90% |
| Assignment accuracy (volunteer and student satisfaction) | > 80% positive feedback |
| Reduction in school dropout | Downward trend |

### Quality Assurance Strategy
- **Unit tests** for all algorithm functions
- **Integration tests** across different scenarios
- **Load testing** with simulations targeting 100+ concurrent users
- **Usability testing** with pilot sessions with UD university students
- **Continuous monitoring** with automatic alerts for error rates and long response times
